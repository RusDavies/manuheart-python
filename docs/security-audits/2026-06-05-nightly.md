## Finding: publish workflow lacked explicit tag/version alignment check

- Severity: Medium / release-supply-chain hardening
- Evidence: `.github/workflows/publish.yml` built and published release artifacts on non-prerelease GitHub Release publication but did not explicitly fail when the release tag differed from `pyproject.toml` project version.
- Impact: A mistaken release tag could publish an artifact whose package version does not match the GitHub Release identity, increasing operator error and provenance ambiguity risk.
- Fix: Added a release-only workflow step that reads `pyproject.toml`, expects `v{project.version}`, and fails the build before artifact publication if the GitHub Release tag differs.
- Regression guard: Added `scripts/check_publish_workflow_policy.py` to assert trusted-publishing controls and tag/version-alignment snippets remain present.
